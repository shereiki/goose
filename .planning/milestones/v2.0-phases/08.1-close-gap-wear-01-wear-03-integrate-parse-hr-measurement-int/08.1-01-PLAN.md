<objective>
Make HR monitor (0x2A37 standard GATT) data flow end-to-end into the upload payload's `hr` stream. This closes gap WEAR-01/WEAR-03 and fixes CR-02 (device_id filter).

Two coupled changes are required, in this order:

1. **Storage path fix (capture_import.rs):** Today `import_captured_frame_timed` unconditionally calls `parse_frame(frame.device_type, &raw_bytes)`, which rejects any frame not starting with `0xAA`. Raw 0x2A37 HR notifications (e.g. bytes `06 48 ...`) are NOT 0xAA-framed WHOOP frames, so they fail parse and never produce a `decoded_frames` row — they land in `raw_evidence` only. As a result, `decoded_frames_between` (used by the upload bridge) never sees HR monitor frames. A `DeviceType::HrMonitor` special-case must store the raw GATT bytes directly as a decoded_frames row so the upload bridge can read them.

2. **Upload bridge integration (bridge.rs):** In `upload_get_recent_decoded_streams_bridge`, add an HR monitor branch that decodes `payload_hex`, calls `parse_hr_measurement`, parses `captured_at` to a unix-seconds f64, and pushes a grouped `{ts, bpm, rr_intervals}` entry to the `hr` Vec. Also implement the CR-02 device_id filter.

Purpose: Without change (1), change (2)'s branch is unreachable — this is the root cause the original Phase 8 work missed.
Output: HR monitor frames stored as decoded_frames rows; upload payload `hr` stream populated with bpm + rr_intervals; real device_id filtering.
</objective>

<plan_deviations>
The executor MUST apply these deviations from CONTEXT.md. They were verified against the actual codebase during planning and override the literal CONTEXT.md instructions while honoring their intent.

1. **chrono is NOT a dependency (overrides D-03 literal text).** CONTEXT.md D-03 says to use `chrono::DateTime::parse_from_rfc3339`. There is no `chrono` in `Rust/core/Cargo.toml` and `bridge.rs` line 3175 explicitly states "Simple ISO-8601 formatting without chrono dependency." DO NOT add chrono. Honor D-03's intent (use `captured_at`, parse ISO-8601 to f64) with a dependency-free parser — write the inverse of the existing `chrono_from_unix` / `days_to_ymd` helpers. The `captured_at` format produced by the codebase is `YYYY-MM-DDTHH:MM:SS.mmmZ` (see `chrono_from_unix` in bridge.rs ~line 3163). Note: "Using a dependency-free ISO-8601 parser per existing no-chrono codebase pattern (D-03 suggested chrono, which is not available)."

2. **HR monitor frames do NOT currently reach decoded_frames (extends D-04/D-06 scope minimally).** CONTEXT.md assumes the bridge can iterate `decoded_frames` rows where `device_type == "HR_MONITOR"`. Verified: those rows are never created because `parse_frame` rejects non-0xAA bytes. A `capture_import.rs` HrMonitor special-case is REQUIRED for the bridge branch to be reachable. This stays within the "fix the upload pipeline gap" scope (D-04) — it is the storage half of the same gap.

3. **CR-02 device_id source is device_model, not a new column (resolves D-06's open question).** Neither `decoded_frames` nor `raw_evidence` stores a per-frame BLE peripheral UUID. `raw_evidence.device_model` holds the sanitized BLE device name the iOS layer sends (see GooseUploadService.swift line 177, which uses the sanitized name as both device_type and device_class). Filter by `device_model` joined from `raw_evidence`. NO migration, NO new column — this satisfies D-06's "use the simplest correct approach."
</plan_deviations>
