## Phase Boundary

Integrate `parse_hr_measurement` (already implemented and tested in `heart_rate_gatt_protocol.rs`) into `upload_get_recent_decoded_streams_bridge` in `bridge.rs` so that HR monitor upload payloads contain decoded hr/rr stream data. Also fix CR-02 (device_id filter currently empty). Scope is limited to the Rust bridge method — no UI changes, no schema migration required.

**What this phase delivers:**
1. `upload.get_recent_decoded_streams` returns non-empty `hr` and `rr` streams for HR monitor frames
2. RR intervals are delivered in grouped format: `{ts, bpm, rr_intervals: [f64, ...]}` per HR reading (user decision D-02)
3. `device_id` filtering is implemented in the loop (CR-02 fix)

**Out of scope:** HR monitor scan UI (W-3), schema migration, changes to GooseUploadService, changes to the iOS layer.

</domain>

<decisions>
## Implementation Decisions

### RR Stream Format
- **D-01:** RR intervals are delivered **grouped with HR**, not as separate entries. Each HR reading from an HR monitor produces one entry in the `hr` stream: `{"ts": f64, "bpm": u16, "rr_intervals": [f64, ...]}`. The `rr` stream Vec remains available for WHOOP data; HR monitor RR data travels inside the `hr` entries.
- **D-02:** The `rr` top-level stream stays as a `let mut rr` Vec. HR monitor reads do NOT push to the top-level `rr` stream — they embed `rr_intervals` inside the `hr` entry. This keeps the schema backward-compatible for WHOOP data that uses the separate `rr` stream.

### Timestamp
- **D-03:** Use `captured_at` from `DecodedFrameRow` as the unix timestamp for HR monitor samples. `captured_at` is an ISO-8601 string — parse it with `chrono::DateTime::parse_from_rfc3339` (already used in the codebase) and convert to `f64` unix seconds.

### Scope
- **D-04:** This phase fixes ONLY the upload pipeline gap (parse_hr_measurement integration + CR-02 device_id filter). No scan UI, no iOS changes.
- **D-05:** Scan UI (W-3) is explicitly deferred to v3.0 milestone.

### CR-02: device_id Filter
- **D-06:** Implement real device_id filtering in the existing `if !args.device_id.is_empty()` block. The `evidence_id` in `DecodedFrameRow` encodes the capture session. Check if the store can look up the peripheral UUID from `evidence_id`, or fall back to storing `device_id` per frame row. Use the simplest correct approach — if `evidence_id` doesn't encode the BLE peripheral UUID directly, add a `device_id` column to the decoded_frames table via a migration (or use an existing approach consistent with how captures are tracked).

### Claude's Discretion
- Error handling: if `parse_hr_measurement` returns `Err` for a stored HR monitor frame, skip the frame silently (no crash, no warning in the payload). Log a counter via `warnings_json` pattern if convenient.
- If adding a `device_id` column to decoded_frames requires a migration, apply the standard migration pattern from `store.rs` (add-if-not-exists approach used elsewhere).

</decisions>

## Existing Code Insights

### Reusable Assets
- `parse_hr_measurement(&[u8]) -> Result<HrMeasurement, String>`: public function, already registered in `lib.rs`. Call directly.
- `hex::decode(hex_str)`: already a dependency (`hex = "0.4"` in Cargo.toml). Use to convert `payload_hex` back to bytes.
- `chrono::DateTime::parse_from_rfc3339`: used elsewhere in bridge.rs for ISO-8601 → unix timestamp conversion. Use for `captured_at`.

### Established Patterns
- Frame loop in `upload_get_recent_decoded_streams_bridge`: match on `parsed` (the `ParsedPayload`) via `serde_json::from_str`. For HR monitor frames, `parsed` will be `None` or a failed parse (0x2A37 bytes don't match WHOOP frame format). Add an `else if frame.device_type == "HR_MONITOR"` branch after the match block.
- `DeviceType::HrMonitor` stored as string `"HR_MONITOR"` in `device_type` column (via `device_type_name`). Use string comparison `frame.device_type == "HR_MONITOR"` in the loop.

### Integration Points
- `upload_get_recent_decoded_streams_bridge`: add HR monitor branch after the `match parsed { ... }` block
- `rr` Vec: needs `let mut rr` (currently `let rr` — immutable). Change to `let mut rr` to keep consistency even if HR monitor RR data goes inside `hr` entries.
- New test in `Rust/core/tests/bridge_tests.rs`: insert an HR monitor frame into a test DB via the capture bridge, then call `upload.get_recent_decoded_streams`, assert `hr[0].bpm` is populated.

</code_context>
