<objective>
Add a `#[cfg(target_os = "android")] pub mod android` module at the bottom of `Rust/core/src/bridge.rs` that exposes the existing `handle_bridge_request_json` function as a JNI-callable entry point: `Java_com_goose_core_GooseBridge_handle`. The shim does NOT implement any new logic — it is a pure calling-convention adapter (JString in → JSON string out).

After this plan, `cargo ndk -t arm64-v8a build --release --lib` must succeed and `nm -D` on the resulting `.so` must show `Java_com_goose_core_GooseBridge_handle`.
</objective>

<threat_model>
- T-01: JNI function panics → JVM crashes (unrecoverable from Kotlin/Java). The shim MUST catch all error cases and return error JSON instead of panicking.
- T-02: String conversion failure for non-ASCII JSON → JNI method returns null jstring. Handled by returning error-encoding JSON string.
- T-03: Null return from new_string() → null pointer dereference on JVM side. Handled by returning std::ptr::null_mut() as last resort (JVM interprets as null String — recoverable).
Severity: MEDIUM (crash risk if panic mitigation is omitted). This is the only plan with meaningful runtime security concern.
Block threshold: HIGH. BLOCKER if panic path is left unhandled.
</threat_model>

      At the very end of `Rust/core/src/bridge.rs` (after the last function `goose_bridge_free_string` and any test modules), append the following module:

      ```rust
      #[cfg(target_os = "android")]
      pub mod android {
          use std::ptr;
          use jni::objects::{JClass, JString};
          use jni::sys::jstring;
          use jni::JNIEnv;

          /// JNI entry point for com.goose.core.GooseBridge.handle(String) -> String.
          ///
          /// Converts the Java string to a Rust str, delegates to the existing
          /// handle_bridge_request_json dispatch function, and returns the response
          /// as a new Java string. Never panics — all errors are returned as JSON.
          #[unsafe(no_mangle)]
          pub extern "system" fn Java_com_goose_core_GooseBridge_handle(
              mut env: JNIEnv,
              _class: JClass,
              request_json: JString,
          ) -> jstring {
              let request = match env.get_string(&request_json) {
                  Ok(s) => s.to_string_lossy().into_owned(),
                  Err(_) => {
                      return env
                          .new_string("{\"ok\":false,\"error\":\"jni_string_conversion_error\"}")
                          .map(|s| s.into_raw())
                          .unwrap_or(ptr::null_mut());
                  }
              };
              let response = super::handle_bridge_request_json(&request);
              env.new_string(response)
                  .map(|s| s.into_raw())
                  .unwrap_or(ptr::null_mut())
          }
      }
      ```

      Key points:
      - `super::handle_bridge_request_json` is accessible because the android module is INSIDE bridge.rs (same file), and `handle_bridge_request_json` is declared `pub fn` at line 2060
      - `extern "system"` is the correct ABI for JNI on Android (not `extern "C"`)
      - `#[unsafe(no_mangle)]` matches the existing pattern used by `goose_bridge_handle_json` in this file (Rust 2024 edition requires the `unsafe` qualifier on no_mangle)
      - Do NOT add `#[no_mangle]` without `unsafe` — the codebase uses `#[unsafe(no_mangle)]` throughout
      - The error fallback returns null_mut() (not panic) — JVM receives null String, not a crash
    </action>
